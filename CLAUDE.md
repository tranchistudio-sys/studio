# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**Amazing Studio** — a wedding photography studio & dress rental management system for a Vietnamese business. UI language is Vietnamese. This is a pnpm monorepo with a React frontend and Express backend.

## Development Commands

Run these in **two separate terminals**:
```bash
pnpm dev:api    # Express API on port 3000
pnpm dev:web    # Vite frontend on port 5000
```

Other common commands (run from root or within the package):
```bash
pnpm build          # Build both API and frontend
pnpm build:ci       # Full CI build with typecheck
pnpm typecheck      # TypeScript check across all packages
```

Within `artifacts/api-server/`:
```bash
pnpm test                          # Run all backend tests
pnpm vitest run src/routes/bookings.test.ts   # Run a single test file
```

Within `artifacts/amazing-studio/`:
```bash
pnpm test                          # Run all frontend tests
pnpm vitest run src/lib/attendance-messages.test.ts  # Run a single test file
```

Within `lib/db/`:
```bash
pnpm push       # Apply Drizzle schema migrations to DB
```

Within `lib/api-spec/`:
```bash
pnpm codegen    # Regenerate React Query hooks and Zod schemas from OpenAPI spec
```

## Monorepo Structure

```
artifacts/
  api-server/       # Express 5 backend (@workspace/api-server)
  amazing-studio/   # React 19 + Vite frontend (@workspace/amazing-studio)
lib/
  db/               # Drizzle ORM schema & DB connection (@workspace/db)
  api-spec/         # OpenAPI spec + Orval codegen config
  api-client-react/ # Generated React Query hooks (do not edit manually)
  api-zod/          # Generated Zod schemas (do not edit manually)
scripts/            # Utility & seed scripts
```

**Never edit `lib/api-client-react/` or `lib/api-zod/` directly** — they are generated by `pnpm codegen` in `lib/api-spec/`.

## Architecture

### Data Flow
```
DB (PostgreSQL) → Drizzle ORM (lib/db) → Express routes (api-server)
  → OpenAPI spec (lib/api-spec) → Orval codegen
  → React Query hooks (api-client-react) → React pages/components (amazing-studio)
```

Vite dev server proxies `/api` to Express on port 3000.

### Backend (`artifacts/api-server/src/`)
- `index.ts` — entry point, Express app setup, CORS, SSE, auth middleware
- `routes/` — 40+ REST route files organized by domain:
  - **Core**: customers, bookings, payments, dresses, rentals
  - **HR/Payroll**: staff, attendance, salaries, job-earnings
  - **Finance**: `revenue/` (split into helpers + sub-files), expenses, fixed-costs, accounting
  - **Post-production**: photoshop-jobs
  - **AI**: claude-sales-assistant, ai-scripts, fb-inbox
  - **Notifications**: notifications (SSE), push-subscriptions (Web Push)
- `migrations.ts` — all DB migrations in one file (~71KB), append-only
- `lib/` — shared backend utilities (auth, GCS storage, background jobs)

Revenue routes are split into `src/routes/revenue/` subdirectory; `routes/revenue.ts` is a re-export shim for backward compatibility.

The build script (`build.mjs`) runs `scripts/check-duplicate-functions.mjs` to prevent duplicate top-level declarations — a guard added after a real incident.

### Frontend (`artifacts/amazing-studio/src/`)
- `pages/` — one file per route (~58 pages)
- `components/` — reusable UI components; `components/ui/` wraps Radix UI primitives
- `lib/` — utilities: formatters, lunar calendar, validation helpers
- `hooks/` — custom React hooks
- `contexts/` — `StaffAuthContext` (auth state, JWT, role)
- `App.tsx` — Wouter router setup

### Database (`lib/db/src/schema/`)
24 schema files. Key tables: `bookings`, `customers`, `staff`, `payments`, `booking_items`, `photoshop_jobs`, `attendance`, `fixed_costs`, `push_subscriptions`, `quotes`, `staff_cast_rates`.

### Authentication
- JWT stored in `localStorage` key `amazingStudioToken_v2`
- Roles: `admin` / `staff`
- Staff default password: their phone number (bcryptjs hashed)
- Admin username: `tranchi`

## Key Domain Concepts

**Bookings**: Can be single-service or multi-service contracts. Multi-service = one parent booking (abstract) + child bookings. Deleting parent cascades to children.

**Revenue calculation** (v2): `realProfit = collected_payments − staff_cast_costs − direct_expenses − operating_expenses`. Fixed costs (`fixed_costs` table, route `/api/fixed-costs`) auto-apply to every month in the filter window.

**Quotes (Báo giá)**: Draft quotes that don't occupy calendar. Converting a quote to a booking is atomic + race-safe (DB transaction + `SELECT FOR UPDATE`). `booking.totalAmount = quote.finalAmount` (deductions already applied); `booking.discountAmount = 0` to avoid double-deduction.

**Staff Colors**: Calendar bookings are color-coded by the **sale staff** (`assignedStaff.sale`). 10-color `STAFF_PALETTE`. Staff colors stored in `staff.color` DB column.

**Photoshop Jobs (Hậu kỳ)**: 7 statuses (`chua_nhan`, `dang_pts`, `da_pts`, `da_fix`, `da_gui_in`, `xong_show`, `tam_hoan`). Progress = `done / (total_photos + extra_photos_requested)`. Incident costs sync to `booking_items` with `type="incident"`.

**Notifications**: SSE for in-app real-time; Web Push (VAPID + Service Worker `sw.js`) for out-of-browser push. Admin receives all; staff receives only their own jobs.

## Conventions

- **Language**: All UI text, comments, and error messages are in **Vietnamese**.
- **File naming**: kebab-case (`photoshop-jobs.ts`), components PascalCase (`StaffAvatar`)
- **DB tables**: snake_case; Drizzle schema files in `lib/db/src/schema/`
- **API validation**: Zod schemas on all endpoints
- **State**: React Query for server state; `react-hook-form` + Zod for forms; Context for auth
- **Routing**: Wouter (client-side)
- **Dates**: `date-fns` with Vietnamese locale; lunar calendar via custom `convertSolarToLunar`
